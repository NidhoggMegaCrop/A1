Background

University Course Catalogue

A university maintains a catalogue of courses. Each course has a unique course code and a name, and is worth a certain number of credit points. Students and administrators need to add courses, remove courses, search the catalogue, list courses in code order, find courses in a code range, and answer indexing queries efficiently.

In this assignment, you will implement a Course Catalogue ADT. The catalogue will first be represented using a simple binary search tree and will later be upgraded to a height-balanced binary search tree. You will also augment the tree with additional data to support indexing operations and construct a balanced catalogue efficiently.

Course Representation

Each course is represented by the following structure:

struct course { 
	int code; 
	char name[MAX_COURSE_NAME]; 
	int creditPoints; 
};

Course codes are positive integers and are unique within a catalogue. Course names are non-empty strings whose length is less than MAX_COURSE_NAME. Credit points are positive integers.

Course Catalogue ADT

The Course Catalogue is an abstract data type. Users of the ADT interact with it only through the functions declared in Catalogue.h. The implementation details are hidden from ordinary users, although the supplied tests include CatalogueStructs.h so that the internal representation can be checked.

Data Structures

The Course Catalogue ADT must be implemented using a binary search tree.

You must use the supplied structs and fields as follows:

  • You must use struct node for the binary search tree nodes
  • The courses in the catalogue must be stored in the course fields of each node
  • The left and right pointers should be used to connect a tree node to its left and right subtrees respectively. These pointers must not be used for any other purpose.
  • The tree field of the catalogue struct must point to the root of the binary search tree

Please note that all your code for implementing the Course Catalogue ADT must be placed in Catalogue.c, except the struct definitions which should be in CatalogueStructs.h.

Part 2: Course Queries

In Part 2, you will implement some operations for querying courses.

Note that there is no specified time complexity requirement for these operations, but to get full marks, you must implement them as efficiently as possible. You are also required to analyse the time complexity of your solutions.

Note: None of these operations should modify the given catalogue.

Function Signature/Description
CatalogueClosest 
struct course CatalogueClosest(Catalogue catalogue, int targetCode);
Returns a course whose code has the minimum absolute difference from targetCode. On a tie, return the course with the smaller code. Assumes the catalogue is non-empty.
CatalogueLevelOrder 
int CatalogueLevelOrder(Catalogue catalogue, struct course courses[]);
Stores all courses in the given courses array in level-order and returns the number of courses stored. Your implementation should use a queue-based traversal. Any helper data structure needed for this traversal must be implemented inside Catalogue.c.
CatalogueGetRange 
int CatalogueGetRange(Catalogue catalogue, int lowerCode, int upperCode,
                      struct course courses[]);
Stores all courses with codes in the range [lowerCode, upperCode] (inclusive) in ascending order in the given courses array and returns the number of courses stored. Assumes that the array is large enough to store all the courses. Assumes that lowerCode is less than or equal to upperCode.

Examples

Suppose the catalogue contains the course codes 1511, 1521, 2521, 3121 and 3311, arranged like so:

  • CatalogueClosest(catalogue, 2500) should return the course with code 2521.
  • CatalogueClosest(catalogue, 2021) should return the course with code 1521, because there is a tie between 1521 and 2521, and in the event of a tie, the course with the smaller code should be returned.
  • CatalogueLevelOrder(catalogue, courses) should store the courses in the courses array in this course code order and return 5: 
    [1521, 1511, 3121, 2521, 3311]
  • CatalogueGetRange(catalogue, 1500, 3000, courses) should store the courses 1511, 1521 and 2521 in the courses array and return 3.

Analysis

For each of the operations above, determine the worst-case time complexity of your implementation, and write your answers in analysis.txt along with an explanation of each answer.

Your time complexities should be in big-O notation and should be in terms of nn and/or hh, where nn is the number of courses in the catalogue, and hh is the height of the underlying tree. In addition, for CatalogueGetRange, your time complexity may also be in terms of kk, where kk is the number of courses returned.

Part 3: Balanced Catalogue

A binary search tree is only guaranteed to be efficient if it is balanced. In this part, you need to update your implementation so that every catalogue uses a height-balanced binary search tree.

After you have updated your implementation:

  • CatalogueInsert and CatalogueDelete must have a worst-case time complexity of O(logn)O(log⁡n).
  • CatalogueFind and CatalogueClosest must have a worst-case time complexity of O(logn)O(log⁡n).
  • The underlying tree must remain height-balanced after every insertion and removal.
  • Any catalogue returned by later parts must also be height-balanced.

Part 4: Indexing Operations

In this part, you will implement some indexing operations.

The index of a course in the catalogue is the index it would occupy if all the courses were stored in an array sorted by course code, starting from index 0. To implement these operations efficiently, you will need to augment your nodes with suitable data and maintain that data during insertion and deletion.

Function Signature/Description Required worst-case time complexity
CatalogueAtIndex 
struct course CatalogueAtIndex(Catalogue catalogue,
                               int index);
Returns the course at index, or {COURSE_UNDEFINED, "", 0} if index is outside [0,n−1][0,n−1]. 		O(logn)O(log⁡n)
CatalogueIndexOf 
int CatalogueIndexOf(Catalogue catalogue, int code);
Returns the index of the course if it exists, and -1 otherwise. 		O(logn)O(log⁡n)
CatalogueCountLower 
int CatalogueCountLower(Catalogue catalogue, int code);
Returns the number of courses whose code is strictly less than code, whether or not code itself exists. 		O(logn)O(log⁡n)

Examples

Suppose the catalogue contains the course codes 1511, 1521, 2521 and 3121. Then:

  • CatalogueAtIndex(catalogue, 0) should return the course 1511.
  • CatalogueAtIndex(catalogue, 2) should return the course 2521.
  • CatalogueIndexOf(catalogue, 3121) should return 3.
  • CatalogueIndexOf(catalogue, 3311) should return -1.
  • CatalogueCountLower(catalogue, 2500) should return 2 because there are two course codes strictly less than 2500: 1511 and 1521.

Testing

We have provided a main program testCatalogue.c which contains basic test cases. You should examine testCatalogue.c to see what the tests do and how the tests call your functions.

To run these tests, first run make, which compiles your code and creates an executable called testCatalogue, and then run ./testCatalogue.

make
...
./testCatalogue
...

The tests use assert, which is a function-like utility provided by assert.h which takes a condition (which is expected to be true) and terminates the program with an error message if the condition is false. For example, consider the following code:

Catalogue catalogue = CatalogueNew();
CatalogueInsert(catalogue, 1511, "Programming Fundamentals", 6);
assert(CatalogueNumCourses(catalogue) == 1);

The code tests whether CatalogueNumCourses correctly returns 1 after one course has been inserted into the catalogue. If CatalogueNumCourses does not return 1, then the program will terminate with an error message and not run the remaining tests. Note that this does not necessarily mean that CatalogueNumCourses is incorrect - it is possible that CatalogueNew or CatalogueInsert are incorrect.

Since failing a test prevents the remaining tests from running, you may want to temporarily disable a test. You can do this by commenting out the assert statement which runs that test or the function which contains the assert. However, you should remember to uncomment it later so that it can be tested.

The given tests are intentionally very simple. In this assignment, you are required to add new tests for each part of the assignment to check more thoroughly that your code is working as expected, and cover edge cases and code paths that might not be exercised by the given tests. If this is visible to you, you should use the course code 9863 in at least one of your tests. For each new test you add, you must write a comment above it describing what it is testing.

Your tests will be assessed on their thoroughness. To write thorough tests, you should read the specification and your own code to identify different cases and expected behaviours, and write tests for each. We intentionally do not specify the number of tests that you have to write, as it is the thoroughness, not the number of tests, that is being assessed, and it is your task to judge how thorough your tests are.

Note that you do not have to test errors, such as malloc returning NULL. You also do not have to test scenarios in which invalid input is given, such as when NULL is given as a catalogue, as it is the user's responsibility to provide valid inputs.

Frequently Asked Questions

  • Are we allowed to create our own functions? You are always allowed to create your own functions. All additional functions you create should be made static.
  • Are we allowed to create our own #defines and structs? Yes.
  • Are we allowed to modify Catalogue.h in any way? No.
  • Are we allowed to #include any other libraries? No. All the libraries required for this assignment are provided already.
  • Are we allowed to change the signatures of the given functions? No. If you change these, your code won't compile and we won't be able to test it.
  • What errors do we need to handle? You should handle common errors such as NULL returned from malloc by printing an error message to stderr and terminating the program. You are not required to handle other errors.
  • Do we need to free memory in the event of an error? No.
  • What invalid inputs do we need to handle? You are not required to handle invalid inputs, such as NULL being passed in as a catalogue. It is the user's responsibility to provide valid inputs.
  • Are we allowed to share tests? No. Testing is an important part of software development. Students that test their code more will likely have more correct code, so to ensure fairness, each student should independently develop their own tests.
